## 🔴 高优先级（可以马上做）

### 1. 通知记录修复

**问题：** 原生 Android 后台通知（喝水、运动、进食、主动关怀）只出现在系统通知栏，没有存入收件箱。目前只有睡眠提醒因为前台 `autoCheck()` 重新触发而碰巧被记录。

**根因：** `KeepAliveService.showAiNotification()` 直接调用 Android 系统通知 API，没有将内容写回 JS 层的 `notificationLogs` 状态。

**方案：**

- [x] `KeepAliveService` 发送通知时，同时把通知内容（title、body、时间戳）追加写入 `SharedPreferences` 中的通知队列
- [x] `MainActivity` 新增 JS 桥接方法 `drainPendingNotifications()`，返回队列中未读取的通知列表
- [x] `App.tsx` 在前台恢复时（`onNativeHeartbeat` 或 `onResume`）调用桥接读取待处理通知
- [x] 读取后逐条写入 `notificationLogs` 并清除原生队列
- [x] 避免重复：前台 `autoCheck()` 生成的通知已经走了 `showNotification()` → 已入收件箱，不需要再走原生队列

**验收标准：**
- 后台期间原生推送的喝水/运动/进食/主动关怀通知，回到前台后出现在收件箱中
- 不会重复记录

### 2. 删除消息确认弹窗

**问题：** 聊天消息删除没有确认步骤，容易误触。

**方案：**

- [x] 删除操作触发时弹出确认对话框（自定义 Modal）
- [x] 显示「确定删除这条消息？」提示
- [x] 多选删除时提示「确定删除选中的 N 条消息？」

**验收标准：**
- 点击删除后不会立刻删除，需二次确认
- 取消后消息保留

---

### 3. 聊天记忆本地搜索

**问题：** 聊天记录多了之后无法快速找到历史内容。

**方案：**

- [x] 在聊天界面顶部添加搜索按钮，展开搜索框
- [x] 输入关键词后实时过滤当前会话中包含关键词的消息
- [x] 显示匹配计数 (N/M)，上下箭头切换
- [x] 点击切换时跳转并高亮关键词 2 秒
- [ ] 支持跨会话搜索（搜索所有档案）

**验收标准：**
- 输入关键词后能过滤出包含该词的历史消息
- 点击能定位到具体位置

## 🟡 中优先级（需要先理清再动手）

### 4. 长按聊天消息操作菜单

**目标：** 长按消息弹出操作菜单，把现有按钮收进去，避免误触。支持多选。

**方案：**

- [x] 长按消息（500ms）触发操作菜单浮层
- [x] 菜单选项：
  - **复制** — 复制消息文本（用户和 AI 消息都有）
  - **编辑** — 编辑已发送的用户消息（仅自己的消息）
  - **重 roll** — 重新生成 AI 回复（仅 AI 消息）
  - **删除** — 删除单条消息（需确认）
  - **多选** — 进入多选模式
- [x] 多选模式下：
  - 每条消息左侧出现复选框
  - 底部工具栏出现「删除选中」按钮
  - 退出多选模式按钮
- [x] 把现有消息旁的独立按钮迁移到长按菜单中

**验收标准：**
- 长按消息出现菜单
- 多选 → 删除流程顺畅
- 单个消息旁的按钮不再裸露在外

---

### 5. 聊天模式切换（情景模式 vs 简洁模式）

**目标：** 用户可切换「长对话情景模式」和「短对话纯聊天模式」，控制 AI 回复风格。

**方案：**

- [ ] 在设置或聊天界面添加模式切换开关
- [ ] 情景模式：System Prompt 包含世界书设定、角色扮演、情景描写
- [ ] 简洁模式：System Prompt 精简掉情景描写要求，只要简短回复
- [ ] 模式跟随当前聊天档案，不同档案可设不同模式
- [ ] 或者在对话中可随时切换（临时生效）

**验收标准：**
- 切换到简洁模式后 AI 不再输出大段情景描写
- 两种模式回复风格有明显区别

---

### 6. AI 幻觉缓解

**问题：** AI 有时编造不存在的数据和事情。

**方案（只能缓解，无法根治）：**

- [ ] System Prompt 中强调「如果不知道就直说不知道，不要编造」
- [ ] 工具调用结果为空时强制 AI 如实告知用户，不得自行脑补
- [ ] 对用户提出的超出 AI 能力范围的要求，引导用户提供更多信息而不是猜测

**验收标准：**
- 编造频率主观上有所下降（难以量化）

## 🟢 低优先级（远期规划，需要更多思考）

### 7. 记忆系统升级

**目标：** 让 AI 记住跨会话的用户信息，形成用户画像。

**待理清的问题：**
- 记忆以什么粒度存储？（事实片段 / 对话摘要 / 用户属性）
- 什么时候触发压缩总结？（token 阈值？时间阈值？）
- 自动生成的记忆如何审核？（用户可编辑/删除）
- 全局记忆 vs 档案记忆的边界在哪？
- 时间节点如何记录？（记忆链）

**大致方向：**

- [ ] 聊天档案内自动压缩：消息数或 token 数达到阈值后触发后台总结
- [ ] 独立记忆管理页面：展示所有记忆条目，按时间排序，可编辑/删除/手动添加
- [ ] 每条记忆带时间戳，方便区分先后
- [ ] 用户画像定期更新（需在档案设置中手动开启，默认关闭）
- [ ] 「转为全局记忆」按钮：将当前档案记忆提升为全局可用
- [ ] 保留现有手动总结功能
- [ ] 记忆启发式搜索：本地相关性排序（TF-IDF 或更简单的关键词匹配，避免引入重量级 ML 依赖）

---

### 8. 人设切换 & 主脑身份认同

**问题：** 切换人设后，新人格如何看待旧人格留下的记忆和对话？主脑（Overseer）作为更高层级的 AI 如何协调？

**需要决策的问题：**
- 切换人格时，用户是否可以选择「锁定」旧人格的对话不被新人设看到？
- 还是不同人格互相知道对方存在，可以互相吐槽？
- 主脑是否作为「幕后老板」统一管理所有人格，不同人格只是主脑的不同「面具」？
- 记忆归属：哪些记忆是人格专属的，哪些是跨人格共享的？

**大致方向：**
- [ ] 理清主脑与子人格的层级关系
- [ ] 切换人格时提供选项：隔离 / 互通
- [ ] 实现人格隔离时，聊天历史不可见；互通时，新人设能看到旧对话但知道自己不是同一个人

### 10. AI 意愿算法

**目标：** 给 AI 一个内部状态，决定它此刻「想不想」做某件事，而不是永远100%服从。

**问题（还没想清楚）：**
- 意愿值基于什么计算？（用户态度 / 时间 / 任务难度 / 随机因子？）
- AI 「不想干」的时候怎么表现？（拒绝？拖延？讨价还价？）
- 这跟人设系统的关系是什么？
- 会不会让用户体验变差？

---

### 11. 思维链流式输出

**问题：** DeepSeek V4 Pro 的 `reasoning_content` 不能以流式方式输出到 UI，目前是等全部完成后一次性展示。

**难点：**
- 取决于 DeepSeek API 是否支持 reasoning 的 SSE streaming
- 可能需要升级到更原生支持流式 thinking 的 API 版本
- Android 原生桥接的 `OpenAIStreamingBridge` 已经尝试解析 `reasoning_content` delta，Chrome 中也可行，但可能某些提供商不支持

---

### 12. 页面排版重新设计

**问题：** UI 布局有些冗余。

**方向（待具体化）：**
- 信息密度优化，减少不必要的空白和装饰
- 移动端优先，确保单手操作友好
- 配色和字体一致性
